### 2026.7.4

このライブラリでは、モノイドのデータを演算を分けて管理する方針を取っていた。具体的には、演算は Monoid コンセプトを満たすクラスで実装して、データの型には既存の int などを使っていた。
しかし、行列のようなデータ型を自分で定義するケースで、使い勝手がいまいちになることに気づいた。具体的には、A + B のように気持ちいいコードを書くためには結局データ型の方にも演算を持たせたくなって、分離が曖昧になってしまうこと、クラスの命名が被りがちなこと、など。
そこで、データを演算をまとめて持つ方針に設計を変更して、モノイドを「データ型をラップして演算を定めたクラス」で表すことにした。これは前のライブラリと同じ方針。

- `identity` はやや長いので `id` にするのもありかもしれない。でもちょっと恒等写像っぽい。
- 使うときに毎回モノイド型に wrap するのが面倒になる。モノイドを使うデータ構造側で `std::constructible_from<M, T>` のような型 T を使ったインターフェースを提供することで書き心地を緩和する。

### 2026.7.18
operator *= は a = a * b の意味のつもりで Monoid コンセプトに要請していた。
パフォーマンスのことを考えるとこれはあった方が嬉しいが、M が非可換の場合に a = b * a を表現できないのがややイケてない。
a = a * b と a = b * a を対等に扱いたい、また、Monoid コンセプトは最小限で済ませたい、と感じたので operator *= を削除する。パフォーマンス面についてはコンパイラの最適化に期待する。
なお、具体的なモノイドを定義する場合に *= を定義しないことを強制するものではない。
Semiring など他の代数系についても同様に扱う。
